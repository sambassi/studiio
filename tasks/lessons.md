## 2026-04-13

- **2026-04-13** | `fontPx` appliquait `textScale` aux emojis de cartes, alors que l'éditeur utilise des tailles Tailwind fixes (`text-sm`/`text-lg`). Résultat : emojis 2.2× trop gros, cartes 2× trop hautes, débordements visuels. | **Règle** : avant de faire passer une valeur CSS par `textScale`, vérifier dans l'éditeur si la classe CSS source (Tailwind) est scale-aware. Les tailles Tailwind (`text-xs/sm/base/lg/xl`) sont FIXES. Seules les valeurs `style={{ fontSize: N * textScale }}` sont scale-aware.

- **2026-04-13** | Studio Son re-exportait en oubliant `watermarkPosition`, `seqGradients`, `titleSize`, `ctaTypography`, etc. → composer tombait sur ses defaults → mise en page différente du préview éditeur. | **Règle** : quand un callsite de `composeAndUpload` existe en plusieurs endroits (éditeur, Studio Son, Creator, Régénérer calendrier), tous doivent transmettre **exactement les mêmes champs design**. Ne jamais passer un sous-ensemble en pensant « les defaults du composer sont bons » — les defaults ne reflètent pas le choix de l'utilisateur.

- **2026-04-13** | `paintSeqGradient` avait une branche « noColor → paint dark » qui se déclenchait même après qu'un poster soit peint → le poster était effacé. | **Règle** : séparer les responsabilités : `paintSeqBackdrop` gère le choix « fond sombre vs couleur » UNE FOIS, `paintSeqGradient` ajoute l'overlay PAR-DESSUS. Ne jamais mélanger les deux dans un seul helper qui « peut ou non dessiner ».

- **2026-04-13** | `noColorBg: true` dans l'éditeur est une PORTE (gate), pas un « applique partout ». Le vrai prédicat éditeur est `noColorBg === true && noColorSequences.includes(seq)`. | **Règle** : avant de répliquer une règle CSS de l'éditeur en Canvas, lire le code JSX exact (infographie/page.tsx:2928-2946 style). Ne jamais inférer la sémantique d'un flag à partir de son nom.

- **2026-04-13** | Le padding/gap des cartes utilisait le ratio `/320` (viewport 9:16) pour tous les formats → en 16:9 (viewport 512), les ratios étaient 60 % trop grands ; label/value/desc utilisaient `fontPx(N * textScale)` correct mais `cssPx(N)` ne respectait pas le viewport. | **Règle** : introduire des helpers `fontPx(cssPx)` (scale-aware) et `cssPx(pxAt320)` (padding/gap) qui divisent par `editorViewportPx = isReel ? 320 : 512`. Ne jamais hardcoder `/320`.

- **2026-04-13** | L'utilisateur rapportait « rien a changé » alors que le composer était à jour. Cause : le post stocké en DB avait été rendu par v_N-1 et nécessitait un click Régénérer pour picker le nouveau code. | **Règle** : bumper `COMPOSER_VERSION` à chaque fix visuel du composer + afficher le bouton Régénérer dans le calendrier quand `meta.composerVersion !== CURRENT_COMPOSER_VERSION`. Les vidéos en DB ne se mettent pas à jour toutes seules.

- **2026-04-13** | Le clic Play dans Studio Son déclenchait `play() → pause()` immédiatement. Cause : un useEffect hérité pausait la vidéo quand la séquence active ≠ 'video' (conçu pour l'ancien mode HTML). | **Règle** : quand on change un flux (HTML rebuild → vidéo full montage), auditer tous les `useEffect` qui pausent/jouent `videoRef` et les garder spécifiques au mode où ils sont pertinents. Ne jamais supposer qu'un effet inactif n'aura pas d'effet secondaire.

- **2026-04-13** | Le batch x10 génèrait 10 posts avec les mêmes cartes parce qu'on envoyait le même topic à Claude chaque fois (convergence vers la réponse canonique). | **Règle** : pour forcer de la variation LLM, ajouter un « angle » différent + un nonce aléatoire au prompt par itération. Le nonce aléatoire évite aussi les réponses mises en cache.

- **2026-04-13** | `Math.round(w * (X / 320) * textScale)` inline était dispersé dans 20+ endroits de `video-composer.ts`. Impossible de changer le viewport ou le scale sans réécrire partout. | **Règle** : introduire des helpers (`fontPx`, `cssPx`, `fixedFontPx`, `paintSeqBackdrop`, `paintSeqGradient`) dès qu'on répète un pattern 3+ fois. Les helpers auto-centralisent les bugs.

## 2026-04-28

- **2026-04-28** | `/creer` POST `/api/posts` envoyait toujours `platforms: []` et `status: 'draft'`, alors que le cron `/api/cron/publish` ne traite QUE les posts en `status='scheduled'`. Résultat : les posts créés depuis l'éditeur n'étaient jamais publiés et restaient bloqués en brouillon, sans aucun feedback à l'utilisateur. Le calendrier avalait aussi silencieusement les erreurs serveur (`console.error` only) → l'utilisateur cliquait « Programmer » et rien ne se passait. | **Règle** : (1) tout flux qui crée un post destiné à la publication doit collecter les plateformes au moment de l'export et set `status='scheduled'` quand au moins une plateforme est cochée — sinon le cron ne ramassera jamais le post. (2) Les `fetch` côté client qui matérialisent une action utilisateur (programmer, sauvegarder, publier) doivent toujours surfacer les erreurs réseau ET les `data.success === false` via `alert(...)` — pas juste `console.error` qui reste invisible.

- **2026-04-28** | L'export bureau de `/creer` figeait `posterUrl = pexelsPhotos[0]` ; un fix intermédiaire avait introduit un fallback `(pexelsPhotos[0] || null)` quand `selectedPhotoIndex < 0`. Le chemin batch/calendrier cyclait `(selectedPhotoIndex + b) % N` — ce qui donne `(-1 + 1) % N = 0` quand l'utilisateur a désélectionné, donc l'iteration 1 utilisait quand même `pexelsPhotos[0]`. Résultat : l'utilisateur reçoit une photo qu'il n'a jamais choisie alors qu'il voulait juste le gradient. | **Règle** : l'utilisateur ne doit JAMAIS recevoir une photo qu'il n'a pas choisie. Désélection explicite (`selectedPhotoIndex = -1`) ou absence de photos signifie : afficher le gradient. Pas de fallback automatique vers `pexelsPhotos[0]` qui surprend l'utilisateur. Pattern correct : `(pexelsPhotos.length > 0 && idx >= 0) ? pexelsPhotos[idx] : null`.

- **2026-04-28** | État partagé entre modes (single/batch) : `batchPhotoIndices` était utilisé en priorité sur `selectedPhotoIndex` via `??` — `const photoIdx = batchPhotoIndices[b] ?? selectedPhotoIndex`. Après une session batch x3 (qui peuplait `batchPhotoIndices=[3,5,2]`), repasser en single mode (batchCount=1) ne nettoyait pas `batchPhotoIndices`. Du coup, `batchPhotoIndices[0]=3` écrasait silencieusement le `selectedPhotoIndex` de la nouvelle session single. Le useEffect de troncature (`prev.slice(0, batchCount)`) gardait quand même `batchPhotoIndices[0]` quand batchCount=1, donc le leak persistait. | **Règle** : pour tout état conditionnel à un mode, ajouter un garde explicite `mode === X ? stateA : stateB` plutôt qu'un fallback `??`. Le `??` ne distingue pas « valeur héritée d'un autre mode » de « valeur active du mode courant ». Pattern correct : `batchCount > 1 ? (batchPhotoIndices[b] ?? selectedPhotoIndex) : selectedPhotoIndex`.

- **2026-04-28** | Quand on intègre un nouveau provider (TTS, paiement, social...) à côté d'un existant, ajouter une branche conditionnelle AVANT le code existant — ne JAMAIS modifier le code existant. Si la nouvelle branche échoue, fallback automatique vers le comportement legacy. Vérifier mentalement les 4 cas : (a) ancien provider OK, (b) nouveau provider OK, (c) nouveau échoue → ancien fallback, (d) tout échoue → degradation gracieuse (browser SpeechSynthesis dans le cas TTS). | **Règle** : pattern strict additif `if (newProvider) { try { ... return; } catch { /* fall-through */ } } /* legacy code unchanged */`. Le fall-through doit produire le même type de retour que le legacy pour que le caller ne distingue pas la source.

- **2026-04-28** | Quand on remplace une source de données (ex: `window.speechSynthesis.getVoices()` → `TTS_VOICES`), TOUJOURS valider les anciennes valeurs persistées (localStorage, DB, query params) contre la nouvelle source. Un ID legacy comme `"Microsoft Hortense - French (France)"` ne matchera jamais un ID Edge `"fr-FR-DeniseNeural"` — sans validation, le state initial devient invalide silencieusement. | **Règle** : `const saved = localStorage.getItem(KEY); const valid = saved && SOURCE.some((x) => x.id === saved); const initial = valid ? saved : DEFAULT;`. Aucun chemin utilisateur ne doit dégrader brutalement à cause d'un changement d'infrastructure interne — fallback silencieux vers un défaut sain.

- **2026-04-28** | Filtrage de gender naïf sur SpeechSynthesisVoice : `name.includes('male')` est buggé parce que `'female'.includes('male') === true` — donc une voix nommée "Female" est rejetée comme féminine. Et la majorité des voix navigateur n'ont ni "male" ni "female" dans leur nom (Thomas/Henri/Amélie/Aurélie en français) — donc le filtre ne pouvait JAMAIS retourner une voix male, et toutes les voix passaient comme "féminines". Résultat : Coralie (F demandée) jouait Thomas, Henri (M demandée) jouait Thomas. Tout monde sur la voix par défaut macOS. | **Règle** : pour matcher une voix par gender quand la source ne fournit pas de champ gender (cas browser SpeechSynthesis), utiliser une regex de noms connus par gender, AVEC exclusion mutuelle. Pattern : `wantFemale ? FEMALE_RX.test(name) && !MALE_RX.test(name) : MALE_RX.test(name) && !FEMALE_RX.test(name)`. L'exclusion mutuelle évite que "Frédéric" matche un FEMALE_RX qui contient "fr".

- **2026-04-28** | Microsoft Edge TTS upstream peut retourner du **succès silencieux à 0 bytes** quand une voix est retirée du catalogue (cas observé : `fr-FR-CoralieMultilingualNeural` — 3 tests directs avec msedge-tts, 0 byte chaque fois ; les autres voix Multilingual comme Vivienne marchent encore, donc c'est voice-spécifique). Si la route ne valide PAS que `audioBuffer.length > 0`, elle propage 200 OK avec body vide → le client interprète ça comme "Edge OK" et n'active pas le fallback → l'utilisateur entend rien (ou le browser fallback se déclenche par un autre chemin et joue Thomas). | **Règle** : (1) toute route qui forwarde un payload upstream doit valider que le payload est non-vide AVANT de retourner 200 — sinon retourner 502 avec le voice ID/ressource concernée dans le message. (2) Quand un provider tiers a un comportement de "succès vide", maintenir une liste de "voix/endpoints connus cassés" et les retirer du catalogue UI au lieu de garder l'option qui ne marchera jamais.
- **2026-04-28** | `ctx.fillText` n'a aucune gestion native des sauts de ligne — le `\n` est rendu comme un caractère invisible et le texte reste sur une seule ligne. Quand on permet à l'utilisateur de taper du multi-ligne (textarea), le rendu Canvas doit splitter par `\n` AVANT le word-wrap, sinon l'intention de saut de ligne est silencieusement perdue. | **Règle** : un helper `wrapText` (ou équivalent) qui prépare des lignes pour `fillText` doit toujours commencer par `text.split(/\r?\n/)`, puis word-wrapper chaque segment. Tester avec et sans `\n` pour garantir la rétrocompatibilité (texte sans `\n` doit produire le même résultat qu'avant).
- **2026-04-28** | Une feature visuelle a 3 endroits à câbler dans ce projet : (1) UI/state dans /creer, (2) composer `drawXxx` logic, (3) design payload passé à `composeAndUpload` AT EACH callsite (2 dans /creer + 4 dans /calendar) + le `metadata.design` pour la persistance. Oublier le 3 produit une feature qui semble fonctionner dans l'éditeur (preview live) mais qui n'apparaît pas dans la vidéo finale exportée. PR #85 a posé l'UI + le composer mais oublié de wire `cardsTypography` dans les payloads — résultat : checkbox active, mais export sans gradient. | **Règle** : pour toute nouvelle propriété design, faire `grep "composeAndUpload({"` dans tout le repo et confirmer que la prop est ajoutée à CHAQUE appel. Ne pas se fier à l'aperçu live qui utilise un chemin de rendu différent.

- **2026-04-28** | Pour appliquer un dégradé sur une icône SVG sans toucher l'image source, pré-rendre l'icône en blanc puis utiliser un canvas offscreen avec `globalCompositeOperation = 'source-in'` pour masquer un linearGradient sur les pixels opaques. **Le canvas offscreen est essentiel** — `source-in` sur le canvas principal effacerait toutes les sequences alentour. | **Règle** : pour tout fillStyle gradient sur une image (icône, photo masquée), passer par un canvas offscreen + `drawImage` final.

- **2026-04-28** | Refactor d'une feature visuelle "tout-ou-rien" vers "par-élément" : il faut (a) migrer le composer pour avoir N gradients/styles distincts au lieu d'un seul, (b) passer les N flags dans le payload via une nouvelle shape, (c) GARDER une logique de migration silencieuse pour l'ancien shape (legacy `textGradient: true` → tous les flags `true`) afin que les anciens posts en metadata ne perdent pas leur effet visuel après le bump de COMPOSER_VERSION. | **Règle** : tout refactor de DesignOptions doit avoir un chemin de migration legacy dans le composer (lecture des deux shapes). Sinon les anciens posts régénérés perdent leurs effets visuels.

- **2026-04-28** | Anti-pattern : un `<select>` HTML avec uniquement les noms textuels d'icônes/emojis/couleurs/assets (`<option>Sparkles</option>`...). L'utilisateur ne voit pas la chose qu'il sélectionne. Le picker des cartes (grille catégorisée + recherche + preview) existait déjà dans le projet — mais le panneau CTA avait été expédié avec un dropdown texte. | **Règle** : pour toute sélection d'icône/emoji/couleur/asset, montrer la chose qu'on choisit. Réutiliser ICON_LIBRARY, ColorWheel, MediaLibrary.
- **2026-04-28** | Le canvas-wide gradient `ctx.createLinearGradient(0, 0, w, h)` appliqué à du petit texte (cartes < 1/10 du canvas) donne une couleur quasi-uniforme. La gradient bbox doit correspondre à la bbox de l'élément. Et html2canvas v1.4.1 ne sait PAS rendre `background-clip: text` → bypass le snapshot quand le gradient est on, sinon snapshot capture du texte invisible. | **Règle** : (1) borner le gradient à la bbox de l'élément ; (2) si rendu via html2canvas, bypass quand gradient actif et laisser le composer canvas faire le rendu.

## 2026-04-29

- **2026-04-29** | Toute prop "extra*" qui clone une feature principale (extraTitle vs title, extraSubtitle vs subtitle) doit cloner TOUS les sous-contrôles (scale, gradient, letter-spacing, line-height, bold, italic), pas juste la valeur textuelle. Sinon UX incohérente : l'utilisateur a des contrôles riches sur le principal et un texte minimal sur l'extra, le résultat exporté semble cassé. | **Règle** : quand on ajoute une feature "extra/secondary" qui mime une feature primaire, dupliquer la liste complète des contrôles + persistance + payload composer + rendu canvas. Audit list : (a) états React, (b) save snapshot localStorage, (c) restore initial + loadConfig, (d) dependency array, (e) UI controls panel, (f) live preview style, (g) chaque callsite composeAndUpload (creer + calendar régénérer), (h) composer DesignOptions type, (i) composer drawXxx logic. Skipper UNE de ces étapes = feature partielle qui apparaîtra cassée à l'export ou après reload.
- **2026-04-29** | `<input type="text">` interdit pour tout champ qui doit supporter `\n` — Entrée soumet un form ou ne fait rien selon le contexte. Toujours `<textarea rows={N} style={{ resize: 'vertical' }}>` côté UI + `whiteSpace: 'pre-wrap'` (ou Tailwind `whitespace-pre-wrap`) sur chaque `<p>` du live preview. Et côté composer, `ctx.fillText` ignore les `\n` — utiliser un helper `fillMultilineText(ctx, text, x, y, lineStep)` qui split sur `\r?\n` et stack les lignes, OU `wrapText` qui fait déjà le split avant le word-wrap. | **Règle** : pour un champ multi-ligne, vérifier les 3 couches : (1) UI = textarea, (2) live preview = pre-wrap, (3) composer = wrapText OU fillMultilineText. Manquer une couche = newlines invisibles à un endroit du flux.
- **2026-04-29** | Lors d'un `style={{... opacity: 0.8, ...(cond ? {..., opacity: 1} : {...})}}`, TS warne TS2783 "specified more than once" même si fonctionnellement le spread override. Solution : déplacer l'opacity dans la branche fallback du ternaire (`: { color, opacity: 0.8 }`) plutôt que la déclarer en dehors et la re-spread. | **Règle** : tester avec `npx tsc --noEmit` et fixer les warnings sur les fichiers touchés avant commit, même si `ignoreBuildErrors: true` masque les erreurs au build.
- **2026-04-29** | Calendar HTML rebuild ≠ éditeur ≠ vidéo composer. Quand un post n'a pas de `renderedVideoUrl` (ex: legacy ou regen en cours), la modale calendar tombe sur le HTML rebuild qui doit reproduire EXACTEMENT le rendu du composer canvas, sinon : (a) flash visible si la vidéo se charge à T=1s avec un layout différent, (b) divergence permanente pour les vieux posts. Fields qui doivent être appliqués dans le HTML rebuild en parité avec composer/éditeur : `titleTypography.textGradient + gradColor1/2`, `cardsTypography.{label,value,description}Gradient + gradColor1/2`, `extraTitle + extraTitlePosition + extraTitleTypography` (idem subtitle), `whitespace: pre-wrap` + `wordBreak: break-word` partout où le user peut taper. | **Règle** : pour tout nouveau champ design, l'audit-list inclut le calendar HTML rebuild en plus du composer et de l'éditeur. Pattern : isoler le style gradient dans une `const _gradStyle = hasColors ? { background-clip text + transparent fill } : null` réutilisable, et le spread après le `color: fallback` dans le `<p>` style.
- **2026-04-29** | Cards container width dans calendar : `<div className="px-3" style={{ width: '92%' }}>` avec Tailwind `box-border` rétrécit le contenu de 24px (px-3 = 12px chaque côté) et fait paraître les cartes "tronquées" car le grid intérieur est plus étroit que les `<p>` `whitespace: nowrap`. L'éditeur n'a PAS de padding outside la grille (juste `width: ${cardsSize}%` + `<div className="grid w-full">`). | **Règle** : le wrapper de cartes du calendar ne doit pas ajouter de padding horizontal — il doit miroiter exactement le wrapper de l'éditeur (width % du parent, pas de px-3). Si du padding est nécessaire pour l'esthétique, utiliser une marge interne dans chaque `<div>` de carte, pas sur le container.
- **2026-04-29** | Persistence localStorage : un dep array d'un `useEffect` de save DOIT inclure CHAQUE field listé dans le snapshot — sinon le save ne re-fire pas quand le user change ce field, et la valeur est perdue au prochain refresh même si elle est référencée dans l'objet snapshot. PR #93 a ajouté les champs `extraTitleScale/Gradient/...` au snapshot mais oublié de les ajouter au dep array → toggle gradient = pas sauvé = perdu au refresh. Idem pour `audioMusicUrl/audioVoiceUrl/audioMusicVolume/audioVoiceVolume` qui n'étaient ni dans snapshot ni dans deps. | **Règle** : audit systématique dep array vs snapshot keys avec un `diff <(grep -oE 'KEYS' snapshot) <(grep -oE 'KEYS' deps)` (ou une review visuelle attentive). Pour chaque nouvelle prop persistée : ajouter dans (1) snapshot save object, (2) dep array, (3) load handler. Manquer même une étape = perte silencieuse.
- **2026-04-29** | Toute modification de `className` conditionnelle dans un rendu UI doit préserver INTÉGRALEMENT la className originale quand la condition est fausse. Pattern à risque : `className="font-bold whitespace-pre-wrap ${gradStyle ? '' : 'text-white'}"` — si l'original avait `font-bold text-white truncate`, on a oublié `truncate`. Le `truncate` Tailwind est `overflow-hidden text-ellipsis whitespace-nowrap` ; ajouter `whitespace-pre-wrap` à côté entre en conflit silencieusement (pre-wrap gagne). Résultat : régression sur Full Width label (texte long ne tronquait plus avec ellipsis). | **Règle** : avant de modifier une className conditionnelle, lire la version pré-modif via `git show HEAD~N:file.tsx` et faire un diff visuel. Si la branche fausse de la condition perd des classes, c'est une régression. Pour les classNames mutuellement exclusives (`whitespace-pre-wrap` vs `truncate`), utiliser une ternaire complète : `gradStyle ? 'class-with-pre-wrap' : 'class-with-truncate'` plutôt qu'un mix.
- **2026-04-29** | Quand on ajoute une feature (ex: gradient, multi-line) à PLUSIEURS styles d'un composant (ex: 6 styles de carte), s'assurer que TOUS les styles supportent UNIFORMÉMENT la feature ET les features pré-existantes. PR #93 a ajouté des sauts de ligne aux 5 premiers styles (Compact/Educatif/Stats Bold/Minimal Line/Full Width) mais Text Only avait perdu le rendu d'icône (alors que les autres styles le supportent). Régression silencieuse car le user n'avait pas explicitement testé Text Only dans la PR. | **Règle** : pour chaque feature multi-style, faire un mini-test matrix : feature × style. Vérifier que CHAQUE cellule de la matrice est valide. Pour les styles qui partagent des fonctionnalités communes (icône, gradient, multi-line), les extraire dans un helper réutilisable au lieu de dupliquer/oublier dans chaque branche.
- **2026-04-29** | `Math.round(string)` retourne `NaN` silencieusement. Quand un helper retourne une string CSS (ex: `editorPxToDvh(N)` qui produit `'1.23dvh'`), wrapper avec `Math.round(...)` casse le rendu sans erreur visible. Lucide-react `<Icon size={NaN}>` ne render rien à l'écran. | **Règle** : si une fonction helper retourne une string CSS unit, ne JAMAIS appliquer `Math.round/Math.floor/parseInt` dessus. Soit utiliser le helper directement (TS error si typage strict), soit extraire la valeur numérique via `parseFloat` AVANT de round, puis re-coller l'unité. Pour les composants qui acceptent number ou string size (Lucide), élargir le type pour éviter `as any` et permettre l'usage natural.
- **2026-04-29** | Calendar HTML rebuild diverged de /creer sur des fields cards-spécifiques (`cardsTextScale`, `cardsLetterSpacing`, `cardsFont`) qui n'étaient PAS inclus dans le `metadata.design` payload du POST `/api/posts`. Résultat : calendar voyait `design.cardsTextScale = undefined` → cardsK = 1 → cartes au mauvais ratio si l'utilisateur avait modifié cardsTextScale (ex: 50% → calendar 2× trop gros). Le `composeAndUpload` callsite (site #1) avait bien `cardsTextScale`, mais le payload POST (site #2) et l'export Bureau (site #3) ne l'avaient pas. | **Règle** : toute prop éditeur consommée à la fois par le composer ET par un autre rendu (calendar, library, etc.) doit être listée dans CHAQUE design payload, pas seulement celui du composer. Audit list à chaque ajout de prop : (1) `composeAndUpload` callsites, (2) `metadata.design` POST `/api/posts`, (3) export Bureau JSON, (4) chaque consommateur de `metadata.design` (calendar render, etc.). Manquer un seul des 4 = divergence visuelle entre éditeur et le consommateur affecté.
- **2026-04-29** | Composer `fontPx` divergait de l'éditeur via un multiplicateur `cardsSizeScale = cardsSize / 92` ajouté commit 1666eb3 (Apr 18) avec l'intention "le texte doit grossir avec la taille du container". MAIS l'éditeur garde des fonts FIXES (label 7px / value 9px / desc 6px sur viewport 320px) indépendamment de cardsSize — seul le container change de width. À cardsSize=110 → 1.20×, à 130 → 1.41×, etc. La trace originale dit "card text sizes scale proportionally with cardsSize" — fausse hypothèse. | **Règle** : pour toute formule de scaling dans le composer canvas, vérifier ligne-à-ligne contre la formule de l'éditeur AVANT de modifier. - **2026-04-29** | html2canvas v1.4.1 ne supporte PAS `background-clip: text`.
